❯ can i link the go code into the Rust ?                                        

● Technically yes. For this project, you almost certainly shouldn't. Here's the
  honest breakdown.

  How it would work

  Go can't be linked as a normal library — it brings its entire runtime (GC,
  goroutine scheduler, signal handlers). The mechanism is cgo:

  - go build -buildmode=c-archive → a .a + C header exposing functions you mark
  //export.
  - Rust links that archive via FFI and calls in.
  - You'd marshal JSON strings across a C ABI: Rust serializes an event → *const
  c_char → Go unmarshals, runs the CEL rule, returns a verdict string/code →
  Rust copies it out and frees per cgo's pointer rules.

  So the seam is: a thin C-ABI evaluate(rule, event_json) -> verdict_json shim
  over cel-go.

  Why it's the wrong trade here — four specific costs

  1. It wrecks your distribution story (the decisive one). §16 ships four static
  targets including x86_64-unknown-linux-musl and aarch64-unknown-linux-musl.
  cgo + cross-compilation + musl static linking is famously fragile — each
  target needs a matching C cross-toolchain and a Go cross-build with
  CGO_ENABLED=1. Your clean cargo build --target … matrix becomes a four-cell
  Go+C cross-compile gauntlet. This alone usually kills the idea.
  2. You embed a second language runtime into "a minimal single Rust binary."
  The Go GC and scheduler now live in your process and binary. §8 chose
  single-Rust-binary deliberately.
  3. It contradicts your own brand. blastradius is a tool that shames 
  supply-chain footguns (§16 — the npm shim refuses to fetch-on-install). Adding
  a Go toolchain to your build/CI and a Go runtime to your artifact is exactly
  the kind of thing someone in your demo room will point at.
  4. Signal-handler and safety friction. Go's runtime installs its own signal
  handlers (SIGSEGV/SIGPROF/…), which fight with your ctrlc handler and
  subprocess handling. And it doesn't help the thing you'd hope — CEL still 
  evaluates over raw values, so you're now marshaling raw command/host strings
  across a language boundary into a runtime you don't control, before your
  normalize.rs Layer-1 redaction. Linking makes the value-free boundary murkier,
  not cleaner.

  The only thing you genuinely gain is cel-go's exact, battle-hardened CEL
  semantics versus a less-mature Rust CEL crate. That's a real but narrow win,
  and not worth the four costs above.

  Better ways to get the same value, in order

  1. Separate process (recommended). Run beacon as the Go sidecar/sensor; it
  evaluates rules and emits JSON; blastradius reads it. Clean ABI (JSON over a
  pipe/socket), independent toolchains, single Rust binary preserved, redaction
  at the read boundary. This is the numerator/denominator division of labor from
  before — and it's less coupling than linking, not more.
  2. WASM, if you want it in-process. Compile the Go threat-rules engine to
  WebAssembly and run it via wasmtime embedded in your Rust binary. This is
  actually the most on-brand link option: it keeps a single Rust binary,
  collapses the cross-compile matrix to one target (wasm), and sandboxes the 
  rule evaluator so it provably can't touch fs/network — which aligns with your
  safety story. Caveats: standard Go→wasm carries runtime weight; TinyGo is
  lighter but may not support all of cel-go's reflection. Worth a spike before
  committing.
  3. Reimplement the CEL-over-events leg in Rust using a CEL crate
  (cel-interpreter), adopting beacon's spec (schema.json/VERSION/rule YAML)
  rather than its code. No Go at all; you keep interop via the shared format.
  Cost: the Rust CEL ecosystem is less hardened than cel-go.

  Bottom line

  Don't cgo-link it. The musl/darwin static-binary story and your supply-chain
  brand are worth more than cel-go's exact semantics. If you want beacon's
  maturity, run it as a process (cleanest) or as WASM (in-process + sandboxed,
  and the only "linking" form that fits your safety/distribution constraints).
  Reserve a from-scratch Rust CEL impl for when you only care about the format,
  not the engine.

  Want me to spike one of these — e.g. sketch the JSON process boundary, or test
  whether the cel-go threat-rules engine compiles to WASM cleanly?
